Challenge 4: DevOps

수동 배포 단계는 오류가 발생하기 쉬우며, 대부분 잘 문서화가 이루어지지 않습니다. 이 도전 과제에서는 컨테이너를 구축하여 자동화되고 반복 가능한 방식으로 AKS 클러스터에 배포하는 플랫폼인 Azure DevOps를 소개합니다. 또한, 이 도전 과제를 통해 배포 모니터링을 위해 사용할 수 있는 유틸리티인 Azure Monitor도 소개합니다.

Exercise 1

Azure DevOps는 본래부터 함께 작업하는 하나의 단일 플랫폼 내에서 더 나은 협업, 더 정밀한 계획 및 빠르고 신뢰할 수 있는 서비스 배포를 모두 가능하게 합니다. Azure DevOps는 기존 작업 흐름을 보충할 수 있도록 GitHub 같은 많은 다른 개발 서비스와 자연적인 통합도 제공합니다. 이 도전 과제에서는 Azure DevOps 전체 중 3개의 영역을 사용합니다.

  • Azure DevOps (소스 제어)

    • 클라우드에 호스팅된 무제한 사설 Git Repos를 제공합니다.
    • 변경된 개별 코드 라인까지 완전한 파일 이력과 함께 애플리케이션 및 인프라의 안전한 저장을 가능하게 합니다.
    • 저속 또는 오프라인 접속 상에서 배포를 가능하게 하는 공통 분산 소스 제어 모델(Git)을 지원합니다.
    • Git는 개발자가 동시에 동일 코드베이스에서 작업하는 것을 도우면서 모든 변경에 대해 개발자가 분리된 환경을 생성하는 능력도 제공합니다.

  • Azure Boards (소프트웨어 개발 관리)

    • 소프트웨어 프로젝트에 대한 작업 관리 기능을 제공합니다.
    • Scrum 및 Kanban 작업 흐름에 대한 자체 지원을 포함해 작업 계획을 돕는 여러 도구로 구성됩니다.
    • 작업이 수반하는 것, 예상되는 노력 그리고 더 넓은 비즈니스 목적에 적응하는 방법에 대해 정의하면서 작업 항목의 계층적 관리를 지원합니다.
    • 사용자가 조정할 수 있는 대시보드와 작업 항목에 대한 복잡한 질의응답을 지원합니다.

  • Azure Pipelines (소프트웨어 배포 파이프라인)

    • 테스트와 배포 파이프라인 구축 자동화를 가능하게 합니다.
    • 다양한 언어, 플랫폼 그리고 다른 도구 및 클라우드와 통합을 지원합니다.
    • 파이프라인을 코드로 정의하는 것을 허용해 파이프라인이 항상 코드와 동기화되는 것을 허용합니다.
    • 자체 컨테이너 지원

이 워크샵에는 여러분의 사용자 계정을 위해 이미 구성이 이루어진 Azure DevOps 인스턴스가 있습니다. 이 연습을 통해, 여러분은 Azure DevOps 포털에 로그인하고, 소스 리포지토리를 설정하며 이용 가능한 다른 도구를 조사할 수 있게 됩니다.

Task

  • 실습 크리덴셜로 Azure DevOps 인스턴스(https://dev.azure.com/azuresprint) 로그인합니다.
  • 팀 프로젝트 내 Azure Board, Azure Repo 및 Azure Pipeline을 조사합니다.
  • Azure Repos에 새로운 빈 리포지토리를 생성합니다.

Hints

  • Azure DevOps에 로그인 시 문제가 발생할 경우 코치에게 지원을 요청합니다!

유용한 리소스

  • DevOps 소개 (영문)
  • Azure DevOps 소개 (영문)
  • 프로젝트에 새 Git 리포지토리 만들기

Exercise 2

이전 도전 과제에서는 많은 시간과 노력이 소요되었습니다. 지금까지 했던 모든 작업을 날려버리는 것에 대해 생각하기도 싫을 것입니다! 다행스럽게도 Azure DevOps는 Azure Repos를 통한 소스 제어 기능을 제공해 변경할 때마다 코드 등록을 허용합니다.

이 도전 과제를 통해, 시간에 따른 코드에 대한 변경 추적을 허용하고 다음 연습 시 배포 자동화를 가능하게 하도록 소스 제어 시스템에 작업 중인 파일을 등록할 수 있게 됩니다.

Tasks

  • 이전 연습에서 생성한 리포지토리를 클론합니다.
  • 애플리케이션의 코드, DockerFile 및 디플로이먼트 YAML 파일을 등록합니다.

Hints

  • Git를 모른다면 애플리케이션에 약간의 변경 사항을 적용하고 브랜치 및 풀 리퀘스트를 연습해볼 것을 권장합니다. 자세한 정보는 "유용한 리소스" 섹션에서 확인할 수 있습니다.
  • 권장하는 (폴더) 구조는 다음과 같습니다:
      app
      - v1
        - index.php
      - v2
        - index.php
      manifests
      - deployment.yml
      - services.yml
      azure-pipelines.yml
      Dockerfile

유용한 리소스

  • 주요 DevOps 개념
  • Git을 사용한 코딩

Exercise 3

소스를 제어하는 장소에 코드를 업로드하는 것은 코드 배포를 위해 필요한 모든 파일을 CI/CD 파이프라인이 액세스할 수도 있게 합니다. Azure Pipeline을 이용해 이 파이프라인을 설정할 수 있습니다. 이를 통해 애플리케이션을 수동으로 구축하거나 배포 명령을 kubectl에 입력할 필요 없이 애플리케이션 또는 애플리케이션이 작동하는 환경 갱신에만 집중할 수 있게 됩니다.

이 도전 과제를 통해, 구축 파이프라인을 설정해 컨테이너 이미지를 구축하고, 이 이미지를 Azure Kubernetes Service 클러스터에 배포합니다.

Tasks

  • 새로운 파이프라인을 생성해 컨테이너를 구축하고 Kubernetes 클러스터에 배포합니다.
  • 배포를 실행하고 애플리케이션을 Kubernetes 클러스터에 배포합니다.
  • 새로 배포된 애플리케이션이 올바르게 실행 중인지 확인합니다.

Hints

  • Azure Kubernetes 템플릿을 사용하면 이전 도전 과제를 통해 생성한 디플로이먼트 YAML 파일을 수정해야 합니다.

유용한 리소스

  • Azure DevOps에서 AKS 파이프라인 만들기

Exercise 4

복잡한 시스템 인프라를 관리할 때 중요한 고려사항은 이 시스템의 상태 모니터링 방법입니다. Azure는 Azure Monitor 범위에 있는 도구 모음을 통해 애플리케이션 및 서비스가 어떻게 수행되고 있는지 조사하는 쉬운 방법을 제공합니다.

AKS는 클러스터 상태 모니터링을 위한 다양한 내장 인사이트 및 매트릭스를 갖고 있습니다. 이 연습을 통해, Azure Kubernetes Service의 모니터링 기능을 조사하고 파드가 심각한 장애를 갖는 경우 알려주는 이메일 경보를 생성할 수 있게 됩니다.

Tasks

  • Azure Portal 내 AKS 리소스에서 인사이트, 메트릭 및 로그를 살펴봅니다.
  • 클러스터에서 각 파드 상태를 봅니다.
  • AKS 리소스에 대한 liveness probe가 인사이트/노드////라이브 데이터 보기 (오른쪽에 있는 버튼)에 나오는지 확인합니다.
  • 파드에 대한 ready 비율이 특정 임계치 (예: 90%) 아래에 있을 때 이메일을 보내는 알림을 만듭니다.
  • 새로운 알림이 도착하는지 테스트하기 위해 파드를 중지 또는 삭제해봅니다.
  • 새로운 파드가 재시작 또는 생성이 되어 알림이 사라졌는지 확인합니다.

Hints

  • Azure Monitor의 여러 많은 기능들이 있습니다. 자유롭게 살펴보세요!

유용한 리소스

  • Azure Monitor
  • 컨테이너를 위한 Azure Monitor
